# 2026-06-10 — Leave-one-out sweep: регрессоров НЕТ, все патчи подтверждены

## Методология

Сервер-side harness (`/tmp/genesis_sweep.sh`, nohup): 8 кандидатов по одному
off + контрольный baseline в конце. Каждый цикл: `ab_overrides.env` →
`docker restart` → wait ready → `genesis_full_bench.py --quick --skip-tools
--max-tokens 1024` (историческая методология). Механизм override: `run.sh`
сорсит `ab_overrides.env` ПОСЛЕ хардкод-экспортов — пустой файл = baseline.

Кандидаты выбраны по принципу «hot-path патчи с потенциалом скрытой
TPOT-регрессии»: кастомные Triton-ядра, kernel-тюны, autotune-прунеры.

## Выводы

1. **Регрессоров среди активных патчей НЕТ.** Гипотеза пользователя
   «какие-то патчи раньше давали плюс, а теперь дают регрессию» на
   проверенном множестве hot-path патчей не подтвердилась: каждый
   выключенный патч либо ухудшал TPOT (+0.2…+0.9%), либо был нейтрален.
2. **P67 — обязательный для корректности**: без него TQ+MTP стек выдаёт
   мусор (0/5 quality). «TPOT 3.597» при выключенном P67 — иллюзия на
   сломанной генерации.
3. Суммарный вклад измеренных патчей ≈ +2.5-4% TPOT (если бы выключили все
   wins разом). Малые эффекты (+0.2-0.3%) в пределах bench-шума по
   отдельности, но направление согласовано.
4. Путь к цели 236+ wall_TPS — **добавление новых оптимизаций**, а не
   чистка существующих. Следующий: PN365 (после revert PN204 residue),
   затем PN357 (draft greedy speedup), K=4 эксперимент.

## P18B замечание

P18B ставит TQ decode stage1 `warps=8/stages=3` при PN296-профиле
`max_warps=4/max_stages=2`. Sweep показал что выключение P18B даёт −0.2%
TPOT (т.е. P18B может чуть вредить или нейтрален). Противоречие с
профилем остаётся — кандидат на перевалидацию параметров на SM 8.6
отдельным микробенчем, но НЕ горящий (эффект в шуме).
