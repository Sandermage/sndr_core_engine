# Реальный путь к 156K-256K — финальное архитектурное решение

**Дата:** 2026-05-13
**Контекст:** после полного цикла исследования single-card 156K+

## TL;DR

**256K возможно сегодня на 2×A5000 fp8 long-ctx**. На 1×A5000 — нет
без месяцев работы. Это physics, не лень.

## Что было проверено эмпирически

| Setup | Max context | Honest finding |
|---|---|---|
| 1×A5000 fp8 | **102K** | vllm pre-flight cap |
| 1×A5000 fp8 + VIRT=1 156K | OOM at boot | tensor allocation eager |
| 1×A5000 TQ k8v4 130K | OOM on first req | KV pool занял весь VRAM |
| **2×A5000 fp8 long-ctx 200K** | **proven 225K real prompt tokens** | full quality |
| 2×A5000 fp8 long-ctx 256K | (testing now) | expected pass |

## Почему single-card 256K — это **multi-week project**

vllm architecturally НЕ поддерживает:

1. **Lazy KV allocation.** `_allocate_kv_cache_tensors` берёт
   `max_model_len × per_token_bytes` физической памяти ПРИ BOOT.
   Не on-demand.

2. **Virtual block addressing.** Attention forward читает
   `tensor[block_id]` напрямую через Triton kernel. Block_id ДОЛЖЕН
   быть < tensor.shape[0]. Python-уровневых hook нет — нужно либо
   modify Triton kernels, либо custom attention impl на Python.

3. **Sliding window для Qwen3.6.** Model trained без SWA, конфиг
   `sliding_window: None`. Включение SWA → quality penalty неприемлемый.

4. **Per-sequence preemption на single user.** Vllm scheduler уже умеет
   preempt'ить OTHER request, но "preempt self" требует разделение
   request на virtual chunks которые могут перемещаться между tier'ами
   — chunk scheduler refactor, не drop-in.

5. **Mamba SSM state.** Qwen3-Next имеет 48 mamba layers с per-sequence
   state. PN95 не может offload (мы explicitly exclude в
   `init_mamba_exclusions_from_kv_groups`). Mamba state — hard floor
   на GPU memory.

## 4 пути к extending 156K-256K на single card

### Path 1: **2-card setup (РЕКОМЕНДАЦИЯ)**

Уже работает. fp8 long-ctx config `a5000-2x-27b-int4-long-ctx.yaml`:
- max_model_len 280000
- Proven 200K real (225K prompt tokens)
- Full quality
- Можно production today

**Стоимость:** одна дополнительная A5000 (~1500$).

### Path 2: **TQ3 lossy KV compression**

`turboquant_3bit_nc` — 3 бита на element vs 8 (fp8) = **3.3× больше
context на ту же память**.

- На single A5000: 102K × 3.3 = ~340K theoretical
- Real ограничение: ~150K (workspace + activations)
- Quality penalty: ~4 pp на benchmarks (per TQ paper)
- Известный bench (club-3090 #119): 27B INT4 + TQ3 на 2×3090 = 18/150
  quality на full benchmark (catastrophic) — TQ3 + MTP сейчас **не
  готов production** без P67 multi-query verify kernel который у нас есть

**Стоимость:** 3-5 дней работы + risk quality drop.

### Path 3: **Phase 7 — Genesis virtual addressing (the real fix)**

Полная architectural работа:

**PN97 (DONE — commit b5c3c72)** — cap KVCacheTensor.size к
physical budget. Prevents OOM при VIRT=1 inflation. **Пререкизит.**

**PN98 (TODO ~2 недели)** — block_id translation table в
attention forward path:
- Pre-step: walk seq.block_table, classify physical vs CPU-resident
- Sync promote из L2/L3 для требуемых
- Sync demote oldest GPU blocks (rolling window)
- Rewrite block_table mapping
- Attention forward читает rewritten table → reads valid data

**PN99 (TODO ~1 неделя)** — scheduler-side preemption coord:
- Self-preempt на chunk boundaries
- Save chunk N state в PN95 L2 перед chunk N+1
- Resume для decode на full context

**Performance cost:** каждый decode step может trigger sync PCIe
demote+promote → **decode TPS падает с 90 → 5-15 tps**. Это **3-10×
медленнее** для 156K+ context. Quality preserved.

Это **right path** но требует **3-4 weeks dedicated engineering**.

### Path 4: **Chunked prefill через external orchestration**

Application-layer fix через custom OpenAI endpoint:
1. Принимает >100K prompt
2. Splits на 80K chunks
3. Sequential prefill каждого chunk
4. PN95 captures KV state checkpoint между chunks
5. На decode восстанавливает full state через L2 promote

**Limitation:** attention НЕ может cross-chunk attend в этой схеме.
OK для multi-turn chat / Q&A / agentic. НЕ ok для needle-in-haystack.

**Стоимость:** 1 неделя работы + acceptable только для определённых
workloads.

## Финальная рекомендация

| Priority | Workload | Path |
|---|---|---|
| **production today** | 156K+ stable | **Path 1** (2-card, proven) |
| **single-card now** | ≤100K | Current config (proven) |
| **single-card 130K-150K** | acceptable quality drop | **Path 2** TQ3 (test pending) |
| **single-card 256K stretch** | accept slow decode | **Path 3** Phase 7 (long project) |
| **agent-style** | >>100K but no needle | **Path 4** chunked endpoint |

## Что мы построили чтобы это поддержать

PN95 multi-tier + 15 commits в этой сессии — это **infrastructure
готовый под Phase 7**:

- L1 pinned pool (3-5× faster PCIe DMA)
- Layer-aware cold-first demote
- Batched async prefetch API
- Stream+event pool (non-blocking transfers)
- PN96 allocator intercept (allocation cliff handler)
- PN97 tensor physical cap (Phase 7 prereq)
- Defensive Anchor #14 (CUDA crash → clean error)
- Prometheus exporter (operator monitoring)
- Compression (zstd/lz4) + Disk tier (50 GiB)

Когда Phase 7 (PN98 + PN99) реализован — этой infrastructure
sufficient чтобы 256K на single card работать (slow but real).
