## What v1.0 Phase 1 adds vs v0.5

| Anchor | v0.5 | v1.0 Phase 1 | Function |
|---|---|---|---|

| 3 mamba-init | ✓ | ✓ | `kv_cache_manager.py::__init__` |
| **4 register** | — | **NEW** | `gpu_model_runner.py::initialize_kv_cache` |

The 4th anchor calls `register_kv_caches(kv_caches, kv_cache_groups)`
immediately after vLLM allocates per-layer GPU tensors. The
TierManager records:
- per-layer tensor reference (`_kv_caches_ref`)
- per-layer shape/dtype/device/bytes_per_block metadata (`_kv_caches_meta`)

Phase 2 (next session) uses these refs to do the actual
`cudaMemcpyAsync(gpu_block, cpu_slot)` demote/promote.

## Critical bug fix during this work

**Initial smoke crashed**: TP1 worker died on boot with the EXAMPLE
config's `tier1: cpu, capacity_gib: 40.0`. Root cause: `_CpuSlab`
allocated 40 GiB pinned host RAM (cudaMallocHost) per TM at __init__
time. With 2× TP workers each trying to allocate 40 GiB → 80 GiB
pinned > 64 GiB container memory limit → cudaMallocHost OOM → TP1
killed.

**Fix**: made `_CpuSlab` allocation LAZY — storage is allocated
on first `store()` call instead of at `__init__`. For Phase 1
(no real demote yet), no `store()` is ever called → no allocation
→ no OOM. EXAMPLE config also shrunk to `cpu: 4 GiB` for safer
default.

**Importance**: this is exactly the kind of "hidden cost" that
ships only after live integration. v0.5 didn't catch it because
v0.5 didn't have the 4th anchor that triggers the wider singleton
install path in workers.
