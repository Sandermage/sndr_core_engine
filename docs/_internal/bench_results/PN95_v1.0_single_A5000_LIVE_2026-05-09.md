# PN95 v1.0 — Single A5000 Live Test (2026-05-09)

## TL;DR

PN95 в текущем виде на single 24GB GPU **расширяет workable context на ~50% (32K → 48K)
ДОКАЗАНО эмпирически**, но при этом **режет TPS в 2× из-за overhead** и **всё равно падает по OOM**
на больших контекстах потому что demote — bookkeeping-only, физически GPU memory не освобождает.

Это первый раз когда PN95 demote доказано сработал в production (5 срабатываний на 48K probe).

5 архитектурных gap'ов найдено — для real value-prop нужен **Phase 3** (3 новых anchor'а + интеграция с vLLM BlockPool).

## Setup

- **Hardware**: 1× RTX A5000 24 GiB (CUDA_VISIBLE_DEVICES=0, TP=1)
- **Model**: Qwen3.6-27B-int4-AutoRound (hybrid GDN + 16 attention)
- **vLLM pin**: 0.20.2rc1.dev93+g51f22dcfd (Genesis pinned image)
- **KV dtype**: TurboQuant k8v4
- **Spec decode**: MTP K=3
- **gpu-memory-utilization**: 0.92
- **max-num-seqs**: 1 (single user simulation)
- **max-num-batched-tokens**: 2048
- **chunked prefill**: ON

## Boot ceiling discovery

| Test | max_model_len | gpu-mem-util | Result | vLLM estimated max |
|---|---:|---:|---|---:|
| 1 | 200000 | 0.85 | ❌ ValueError @ _check_enough_kv_cache_memory | (not reported) |
| 2 | 64000 | 0.92 | ❌ Same — 2.23 GiB needed, 2.13 GiB available | **59136** |
| 3 | 56000 | 0.92 | ✅ Boot OK | (within ceiling) |

**Hard ceiling on this hardware = 59136 tokens.**
PN95 не помогает на boot потому что не учитывается в pre-flight check (см. Find #1).

## A/B comparison — same probes both arms

| Probe | PN95 OFF | PN95 ON (TICK_EVERY=5) |
|---|---|---|
| **16K (18.3K real)** | ✅ TPS 0.55, VRAM +1682 MiB | ✅ TPS 0.25, VRAM +1732 MiB |
| **32K (36.6K real)** | ❌ HTTP 500 OOM | ✅ TPS 0.10, VRAM +1740 MiB |
| **48K (54.9K real)** | ❌ Connection reset | ❌ HTTP 500 (но дольше выжил, demote сработал 5 раз) |
| **54K** | not tested (already crashed) | ❌ Container crash |

**Headline:**
- ✅ **PN95 расширил workable context с 30K до 45K = +50%**
- ❌ **PN95 overhead на 16K probe: TPS 0.55 → 0.25 = -55% latency hit**
- ❌ **48K оба упали** — PN95 не доделан до конца

## 5 архитектурных gap'ов (ranked by importance)

### Gap #1 (КРИТИЧЕСКИЙ): boot-time validation не учитывает CPU tier
- **Где**: `vllm/v1/core/kv_cache_utils.py::_check_enough_kv_cache_memory` (line 690)
- **Проблема**: `available_memory` = только GPU; CPU tier из `CacheConfig.tiers` ignored
- **Эффект**: `max_model_len > 59K` падает на boot, PN95 не успевает запуститься
- **Fix**: Anchor #6 (text-patch на pre-flight) + helper `pn95_extra_logical_memory_bytes()`
- **Effort**: 2-3h

### Gap #2 (КРИТИЧЕСКИЙ): demote НЕ освобождает GPU memory
- **Где**: `_pn95_runtime.scheduler_tick` → `tier_manager.demote_block`
- **Проблема**: copy bytes GPU→CPU работает, но `view = tensor[block_idx]` остаётся
  Python-side reference на GPU tensor. PyTorch allocator продолжает считать memory занятой.
- **Эффект**: 5 успешных demote, 0 MiB GPU освобождено → CUDA OOM
- **Fix**: после demote_block надо вызвать `vllm.block_pool.free_block(block_idx)`,
  чтобы vLLM пометил блок свободным и его можно было переиспользовать. Plus
  intercept в `BlockPool.allocate` для прозрачного promote при touch демотированного.
- **Anchor #9** (NEW) + `Anchor #10` (BlockPool integration)
- **Effort**: 6-8h (требует deep-знание vllm v1 block_pool internals)

### Gap #3 (HIGH): physical num_blocks не bounded когда logical inflated
- **Где**: `kv_cache_utils.py::get_num_blocks` + `KVCacheTensor` allocation site
- **Проблема**: если Anchor #6 расширит available_memory, vLLM выделит больше physical
  GPU блоков → physical CUDA OOM при init
- **Fix**: split logical (для scheduler) vs physical (для GPU alloc) num_blocks
- **Anchor #7** (NEW)
- **Effort**: 6-8h (invasive — нужно протащить два-track accounting через kv_cache_utils)

### Gap #4 (MEDIUM): TICK_EVERY=100 слишком высокий для single-stream
- **Где**: `_pn95_runtime.scheduler_tick` (default `GENESIS_PN95_TICK_EVERY=100`)
- **Проблема**: single-stream long prefill делает только ~30 schedule() calls на весь
  request. Tick никогда не достигает 100 → demote check не срабатывает.
- **Эмпирика**: с TICK=5 demote сработал 5 раз; с TICK=100 — 0 раз.
- **Fix**: 
  1. Снизить default TICK_EVERY до 10 OR
  2. Сделать event-driven: проверять не каждые N tick'ов, а перед каждой
     `BlockPool.allocate` когда `n_free < threshold_blocks` (синхронно)
- **Effort**: 1-2h (для опции 1) или 4-6h (для опции 2)

### Gap #5 (MEDIUM): observability gap — TM не доступен из API process
- **Где**: TierManager singleton живёт в EngineCore worker; APIServer process не имеет
- **Проблема**: невозможно сделать `/v1/pn95/stats` HTTP endpoint для оператора
- **Эмпирика**: `docker exec ... python3 -c "tier_manager()"` возвращает None
- **Fix**: 
  1. IPC между EngineCore и APIServer для PN95 stats (sharedm-mem или mp.Queue)
  2. ИЛИ простой: каждые N тиков EngineCore пишет stats в `/tmp/pn95_stats.json`,
     APIServer читает при HTTP запросе
- **Effort**: 3-4h

### Bonus Gap (LOW): demote heuristic — block_idx=0 always
- **Где**: `scheduler_tick` цикл `_TM.demote_block(layer_name, 0)` — всегда нулевой блок
- **Проблема**: это не реальный LRU. Может демотить hot block.
- **Fix**: использовать `_admit_order` LRU tracking + выбирать tail (cold)
- **Effort**: 2h

## Что НЕ работает (Phase 3 must fix)

- ❌ Demote не освобождает GPU memory → инкрементальное использование продолжается
- ❌ Performance penalty -55% TPS на 16K (overhead scheduler_tick + mem_get_info на каждые 5 ticks)
- ❌ Нет HTTP endpoint для real-time PN95 stats наблюдения

## Recommendations

### Сейчас (без Phase 3)

**PN95 НЕ готов для production использования.** Default OFF, опционально для experiments.

Если хочется попробовать — конфиг для single 24GB GPU:
```yaml
genesis_env:
  GENESIS_ENABLE_PN95_TIER_AWARE_CACHE: '1'
  GENESIS_PN95_TICK_EVERY: '10'              # was 100, single-stream slow
  GENESIS_PN95_DEMOTE_FREE_MIB_THRESHOLD: '2048'
  GENESIS_PN95_DEMOTE_BATCH: '8'

vllm_args:
  --gpu-memory-utilization 0.92
  --max-model-len 56000   # ~95% от vllm hard ceiling 59136
```

Ожидать: +50% workable context при -55% TPS. Tradeoff неприемлем для большинства workloads.

### Phase 3 — must-do для real value

**Минимально жизнеспособный set** = Gap #1 + #2 + #3 = 3 новых anchor'а:

| Anchor | Файл | Что делает |
|---|---|---|
| #6 | `kv_cache_utils.py:_check_enough_kv_cache_memory` | расширить available_memory на CPU tier |
| #7 | `kv_cache_utils.py:get_num_blocks + KVCacheTensor alloc` | logical/physical block split |
| #8 | `block_pool.py:get_new_blocks` | sync demote on pool exhaustion |
| #9 | `tier_manager.demote_block` (modify) | call vllm `block_pool.free_block` after copy |
| #10 | `block_pool.py:allocate` (or get_block) | promote-on-touch для virtual blocks |

**Effort**: ~20-25h focused work = 1 long autonomous session или 3-4 нормальные.

### Phase 3 success criteria

- ✅ Boot с `max_model_len=120000` на single A5000 (CPU tier даёт +66K headroom)
- ✅ Single 100K probe completes successfully
- ✅ Demote реально освобождает GPU memory — последующий allocate проходит без OOM
- ✅ A/B vs OFF: max workable context **2-3× больше**
- ✅ A/B vs OFF: TPS overhead **<10%** на normal workload (не bound на limit)
